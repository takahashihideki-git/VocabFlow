# h0 減衰・朝学習者パラメータ検討レポート

**実施日:** 2026-04-08  
**検討テーマ:** h0 の動的制御による SRS 最適化  
**使用設定:** `sessionSize=20` / `maxNewPerSession=5` / `waveSize=50` / `waveUnlockRatio=0.7` / `waveUnlockH=2.0`

---

## 発端：位相同期問題（extend.md #3）

同じ日に大量導入した単語が全部 `h0=1.0` から始まるため、due タイミングが揃ってしまう（位相同期）という問題が指摘された。

**仮説（Opus 4.6 提案）：**

```
h0_adjusted = h0 × decayBase^todayNewCount
decayBase = 0.98,  h0Min = 0.3
```

「今日の N 語目ほど h0 を小さくすることで due タイミングが自然にばらけ、位相同期が解消される。認知負荷の現実をモデルに組み込む設計思想」

---

## 実験 1：h0 decay（0.98/語）の効果検証

**対象:** `scripts/sim_h0decay.js`  
**条件:** まとめ消化（4分間隔×5セッション）vs 分散消化（2.5h間隔×5セッション） × h0固定/減衰  
**期間:** 14日間、5回平均

### 結果

| 条件 | 復習なし回数 | 復習なし率 | Day14学習済み |
|------|------------|-----------|-------------|
| まとめ×h0固定 | 3.6回 | 5.1% | 108語 |
| まとめ×h0減衰 | 5.6回 | 8.0% | 101語 |
| 分散×h0固定  | 0回   | 0.0% | 119語 |
| 分散×h0減衰  | 0回   | 0.0% | 117語 |

### 結論

**h0 decay（0.98）はまとめ消化問題を解消しない。むしろ悪化する。**

分散消化では差が軽微（-2語）。まとめ消化での悪化は、h0 が小さいと `peakH≥2.0` に到達するまでの review 回数が増え、Wave 2 解放が遅れるため。

## 実験 2：「復習なし」の発生メカニズム解剖

**対象:** `scripts/sim_h0decay_debug.js`  
**目的:** なぜ h0 decay で復習なしが増えるかを Session 単位で追跡

### 発見

「復習なしが増える」という現象は h0 decay とは本質的に無関係。まとめ消化の構造的問題：

```
Day 4 以降:
  S1: urgent/due を一気に消化 → 20枚フル稼働
  S2: 残りを消化
  S3〜S5: 全員たった今復習済み、次の due は 5〜9h後 → 空
```

**「復習なし」の正体は「Wave 1 枯渇 → Wave 2 未解放」のギャップ**

- Wave 1 の 50 語を 4 日（20 セッション）で導入完了
- Wave 2 解放には `peakH≥2.0` の語が 70% 必要（35語）
- まとめ消化では 1日1回しか review が回らないため Wave 2 解放が Day 7 前後
- この 3〜4 日間のギャップで、各日 S3〜S5 が空になる

h0 decay はこのギャップを短縮せず、むしろ `peakH` の成長を遅らせて延長する。

## 実験 3：セッション間隔逆算 h0

**対象:** `scripts/sim_h0_interval.js`  
**発想:** 「減衰率自体がセッション間隔に応じて変動すべき。4分後セッションの減衰率は 0.001 レベル」

### 設計

```
h0 = min(baseH0, sessionInterval / retentionFactor)
```

- 4分間隔 → h0 = 0.012日（due = 4分後 → 次セッションに登場）
- 2.5h間隔 → h0 = 0.444日（due = 2.5h後 → 次セッションに登場）
- 6h以上 → h0 = 1.0（現行と同じ）

**まとめ消化はSRS的に「足踏み」**：h が育たない。これは認知的現実の正しいモデル化。

### 結果

| 条件 | 復習なし/21日 | 学習済み | avgH Day21 |
|------|------------|---------|-----------|
| まとめ×h0固定 | 3.0回 | 144語 | 17.38 |
| まとめ×h0逆算 | **16.3回** | 100語 | **11.43** |
| 分散×h0固定  | 0回 | 159語 | 23.61 |
| 分散×h0逆算  | 0回 | 146語 | 21.15 |

### 結論

**「足踏み」効果は正しく機能している（avgH 11.43 vs 17.38）が、副作用が深刻。**

`h0=0.012` から `peakH≥2.0` に到達するには約 8 回の正解が必要（`h0=1.0` なら 2 回）。Wave 2 解放が大幅に遅れ、復習なしが 16 回に急増する。

分散消化への影響は軽微（復習なしゼロを維持、学習語 -13 語）。

## 実験 4：「毎朝 5 セッション学習者」パラメータ最適化

**対象:** `scripts/sim_morning_learner.js`  
**対象ユーザー:** 「毎朝 20 分、5 セッションで英単語を覚えたい」（至極真っ当なユーザー像）  
**学習パターン:**

```
1日1回、朝に 5 セッション（4分間隔）
セッション間隔 ≈ 24時間（セッション群間）
目標: 毎朝 meaningful カード（urgent+due+new）≥ 100枚
```

### 仮説と理論分析

`targetRetention=0.85` 時、翌朝 due になる h の上限 = **4.27日**

| alpha | 翌朝 miss までの正解回数 | h の推移 |
|-------|---------------------|---------|
| 1.30 | 5回でやっとmiss | 1.0→1.04→1.35→1.76→2.28→3.86 |
| 1.50 | 5回 | 1.0→1.20→1.80→2.70→4.05→7.90 |
| 1.75 | 3回 | 1.0→1.40→2.45→4.29 |
| 2.00 | 3回 | 1.0→1.60→3.20→6.40 |

仮説：alpha を小さくすれば h の成長が遅くなり、翌朝 due に留まる review が増える → meaningful 増加。

### スキャン結果（21日間、8回平均）

| alpha | ret | 復習なし | meaningful/SS | Wave2解放 | Day21学習済み |
|-------|-----|---------|-------------|---------|-------------|
| 1.30 | 0.85 | 35回 | 15.4枚 | **解放されず** | 50語 |
| 1.50 | 0.85 | 28回 | 15.2枚 | Day17 | 78語 |
| 1.60 | 0.85 | 7回  | 16.5枚 | Day7  | 114語 |
| 1.75 | 0.85 | 3回  | 16.5枚 | Day6  | 125語 |
| **1.80** | **0.85** | **4回** | **16.6枚** | **Day5** | **129語** |
| **2.00** | **0.85** | **2回** | **16.3枚** | **Day4** | **143語** |

`targetRetention` の変化（0.75〜0.85）は同一 alpha 内での影響が小さく、alpha の方が支配的。

### 逆説的結論

**現行デフォルト（alpha=2.0, targetRetention=0.85）が毎朝学習者にも最良。**

alpha を下げると悪化するメカニズム：

```
alpha ↓ → h の成長が遅い
       → peakH≥2.0 到達に必要な review 回数が増える
       → Wave 解放が大幅に遅延
       → 「daily due は豊富」でも「新語が来ない」
       → 復習なし多発・学習語数激減

alpha=1.3: h が recognition→recall で 1.35 止まり（<2.0）
           → 3回目の review が必要 → 70%×50語=35語が揃うまで Wave 2 解放不可能
```

`Wave unlock` が binding constraint であり、h の成長速度より優先度が高い。

### 「100枚/日」目標の未達

現行デフォルトでも毎朝 meaningful が目標 100枚に届かない（Day21で91枚）。

```
Day1:  50枚（新語のみ、review なし）
Day7:  78枚
Day14: 85枚
Day21: 91枚
```

これはパラメータ調整では解決できない構造的制約。本当の改善レバーは：

- **`waveSize` を大きくする**（例: 50→100語）→ 新語在庫が4日間続く → 朝の全セッションで新語が供給され続ける
- **`maxNewPerSession` を増やす**（例: 5→7語）→ 新語ペースを上げる

## 総括

### 試みたこと

| 試み | 期待した効果 | 実際の結果 |
|------|------------|-----------|
| h0 decay (0.98/語) | 位相同期解消 | 復習なし増加（Wave unlock 遅延） |
| セッション間隔逆算 h0 | 次セッションに due で登場 | 復習なし 5倍増（peakH 成長停止） |
| alpha 引き下げ | daily due 増加 | Wave 2 解放が遅れ逆効果 |
| targetRetention 引き下げ | due 判定の早期化 | alpha ほど効果なし |

### 核心的な発見

**「h の成長を遅くして daily due を増やす」と「Wave を早く解放して新語を供給する」は構造的にトレードオフ。**

Wave unlock が `peakH≥2.0`（70%基準）に依存している以上、h の成長を抑制する施策は必ず Wave 解放を遅らせる。毎朝学習者にとって新語供給は due 語と同等以上に重要であり、Wave 解放が遅れると復習なしが増える。

### 次に検討すべき方向

1. **`waveSize` と `maxNewPerSession` の調整** ✅ **解決済み（2026-04-09）**  
   - 小さくする方向（25語）は逆効果：Wave 枯渇が早まり復習なし急増  
   - **大きくする方向（100語）が正解**：新語在庫が4日間持続し、朝集中学習者の復習なしがゼロに  
   - `scripts/sim_wavesize.js` で検証。`core/config.js` の `waveSize` を 50 → **100** に変更済み

2. **Wave unlock 条件の見直し**  
   `peakH≥2.0`（現行）を review 回数ベースの条件に変更し、h 成長速度から切り離す。

3. **セッション数に応じた自動最適化**  
   1日のセッション数（1セッション vs 5セッション）を検出し、`maxNewPerSession` を動的調整する設計。
