# ベイジアン × SRS — 積極的価値と実装可能性に関する提案書

**対象:** VocabFlow / Word Wave SRSウェーブ方式
**位置づけ:** 本書は仕様書ではなく提案書である。実装の採否はシミュレーションによる検証を経て判断する。
**背景:** Claude Fable 5 によるコードレビューで、ベイジアン層（μ/σ/uncertain プール)がデッドコードであると指摘された。本書はその指摘を受けての削除提案ではなく、**ベイジアン × SRS の本来の価値を正しい実装形で回復する**提案である。

---

## 1. 経緯の総括 — 何が起き、何を学んだか

### 1.1 当初の構想（正しかった）

設計初期、ベイジアンアプローチの価値は次のように定義された。

> 「ベイジアンの強みは不確実性を明示的に扱える点。『この単語は90%覚えていそう』と『50%で不確実』を区別できるので、フィードに何を出すかの判断材料が豊富になる」

すなわち、**不確実性を行動（出題判断）に使う**という構想である。同じ h = 4.0 でも、2回の観測に基づく推定と10回の観測に基づく推定では信頼性が異なる。この洞察は統計的に正当であり、現在も正しい。

### 1.2 実装での縮退（ここで失われた）

実装段階で、この構想は「σ が閾値を超えた語を uncertain プールに入れる」という形に縮退した。さらに：

- σ の更新則が場当たり的だった（正解でも不正解でも σ 単調減少。不正解は「h の推定が外れた」証拠なので本来 σ は増えるべき）
- μ は log(h) を代入するだけでどこからも読まれない死にフィールドだった
- σ0=1.0・閾値1.5・成長0.01/日 という設定では uncertain プール入りの条件が非現実的で、プールはデッドコードだった

「ベイジアン推定」という実在する枠組みの名を借りながら、具体的な更新則はどの文献にも基づかない即興だった。参照先がないため、明らかに誤った更新則（不正解で σ 減）の混入を誰も検出できなかった。

### 1.3 棄却の危機（二番目の誤り）

レビュー指摘を受けた当初の総括は「ベイジアン層は不要だった。HLR ルールベースで十分」だった。これは**実装が偽物だったことを根拠に、構想ごと棄却する**誤りである。

death spiral、位相同期、新語飢餓、トリアージ破綻——本プロジェクトが苦しんだ問題のいくつかは、実は「不確実性を行動に使う」本来の構想が解決し得たものだった（後述）。

### 1.4 教訓

1. **新しい状態変数を導入するときは2つの問いを立てる。** 「既存のデータから導出できないか」「それが死んでいたらどのテストが失敗するか」。σ はどちらにも答えられなかった
2. **洞察が実装で縮退していないか、元の構想と定期的に突き合わせる。** 「不確実性を行動に使う」が「閾値判定で箱に入れる」に縮退したことに、誰も気づかなかった
3. **実在の手法の名を借りた即興は、参照照合を不能にする。** 「ベイジアン風」の見た目は、誤りの検出を妨げる

## 2. 不確実性の正体 — 状態ではなく導出

h の推定の不確実性は、2つの既存データから導出できる。

1. **観測回数（reviewCount）:** 少ないほど h の推定は粗い
2. **最終観測からの経過時間（Δt = currentTime − lastReviewed）:** 長いほど推定の根拠が古い。h = 20日という推定が40日前の観測に基づくなら、その間の干渉や類似語の学習による記憶の変化を反映していない

σ という独立した状態変数を維持する必要はない。必要なときに導出すればよい。

```javascript
// 不確実性の幅（導出関数。状態を持たない）
// 観測が少ないほど、観測が古いほど幅が広がる
function uncertaintyWidth(word, config, currentTime) {
  const obsFactor = config.uncertaintyBase / Math.sqrt(Math.max(1, word.reviewCount));
  const deltaT = currentTime - word.lastReviewed;
  const staleFactor = config.staleGrowth * Math.log(1 + deltaT);  // 経過時間で対数的に増加
  return Math.max(config.uncertaintyFloor, obsFactor + staleFactor);
}
```

なお、旧設計の sigmaDecay（時間経過で σ 増加）は staleFactor として正しい直感だった。誤りは、それを状態変数の更新として持ったことにある。導出関数なら、保存も更新も不要で、常に現在時刻に対して正しい値が得られる。

**提案: μ/σ 状態変数と uncertain プールを削除し、上記の導出関数に置き換える。**

## 3. 活用先1 — トンプソンサンプリングによる位相同期の分散

### 3.1 問題

同じ日に導入された語は h・lastReviewed・reviewCount がすべて揃う。p（記憶保持率）でソートしても区別がつかず、同時に due になり、同時に復習され、同時に h が伸び、また同時に due になる。このクラスタ（位相同期）はドッグフーディングで実際に観測された。

h0 にランダムノイズを加える案も検討されたが、恣意的であり、根拠を持たない。

### 3.2 提案: due 判定時の h サンプリング

点推定の h で決定論的にスケジュールするのではなく、**不確実性の幅の中からサンプリングした h_effective で due を判定する**。

```javascript
// due 判定時（セッション生成ごとに引き直す）
function effectiveH(word, config) {
  const w = uncertaintyWidth(word, config);
  const noise = 1 + (Math.random() * 2 - 1) * w;   // [1-w, 1+w] の一様乱数
  return word.h * noise;
}
```

- **観測が少ない語** → 幅が広い → due タイミングが大きく散る → 位相同期が壊れる
- **観測を重ねた語** → 幅が狭い → ほぼ点推定どおりの安定スケジュール

これはバンディット問題における**トンプソンサンプリング**（事後分布からのサンプリングで行動を選ぶ）の構造であり、「探索と活用の両立」という確立された理論的基盤を持つ。場当たりなノイズ注入とは異なり、**不確実性が大きいものほど大きく揺らぐ**という情報に基づいた分散である。

### 3.3 体験上の意味

「若い記憶はゆらぎ、成熟した記憶は安定する」。同じ日に覚えた単語が、それぞれ違うタイミングでふらっと戻ってくる。機械的な一斉再出題より生き物らしく、学習体験としても望ましい。

## 4. 活用先2 — urgent プールのトリアージ

### 4.1 問題

urgent（p < 0.5）が sessionSize を超えたとき、現状は p 昇順という一次元の優先度しかない。ドッグフーディングで「h=16.6 まで育てた mastered 語（p=0.17）が、ほぼ未学習の h=1.0 語の群れ（p≈0）に埋もれて出題されない」事象が発生した。

### 4.2 提案: 多次元トリアージスコア

```
triage_score = (1 - p) × investment × confidence_penalty

investment        = log(1 + peakH)        // 育てた記憶ほど救済価値が高い
confidence_penalty = 1 + uncertaintyWidth  // 推定が不確かな語ほど確認を優先
```

「忘れかけ度 × 投資量 × 不確かさ」の積。一度育てた記憶の救済と、まだ何も知らない語の導入を区別できる。重みづけはシミュレーションで調整する。

## 5. 活用先3 — 学習者への信頼性フィードバック

### 5.1 問題

「定着 411語」という表示は、内訳の信頼性を区別しない。実際には「10回の復習で確認済みの定着」と「数回の正解で h が跳ね上がっただけの定着」が混在する。

### 5.2 提案: 表示への観測回数の統合

ベイジアン事後分布を実装せずとも、観測数と観測の新しさを見せるだけで信頼性は伝わる。

- Word Wave ポップオーバー: 「記憶強度 16.6日（復習8回・最終確認3日前）」
- 統計表示: 「定着 411語（うち確認済み 380語）」— 確認済み = uncertaintyWidth が閾値以下
- UIラベル仕様書の「確信度: 高/中/低」は、σ ではなく uncertaintyWidth ベースの導出で実装する

---

## 6. 活用先4 — 信頼性ベースの確認モード

### 6.1 受動的表示から能動的モードへ

信頼性の表示は、それを能動的に解消する手段とセットで初めて完結する。「未確認 31語」という表示の隣に、「この31語を確認する」ボタンを置く。

**確認モード:** uncertaintyWidth の高い順（= まぐれの可能性が高い順）にデッキを構成し、集中的にテストする特別セッション。

- 正解 → 観測が増えて信頼性が上がる。h も通常通り更新
- 不正解 → h が修正されて通常の復習ループに戻る
- どちらに転んでも **h の推定が改善する**

### 6.2 期末対策モードとの相乗効果

試験3日前、「定着411語」を信じてよいか。信頼性の低い順に一周すれば、まぐれ定着が炙り出される。学習者は表示を見て自発的に学習計画を調整できる——これが信頼性フィードバックの本来の目的である。

### 6.3 情報利得ベース選択の実用的実装形

初期設計で「将来拡張」とされた情報利得ベースのアイテム選択（テスト結果が推定を最も改善する語を優先する）は、本モードがその実用的な実装形である。uncertaintyWidth の高い語ほど、テスト結果がもたらす情報量が大きい。複雑な情報理論の計算は不要で、導出関数のソートだけで近似できる。

## 7. 実装提案まとめ

| 項目 | 内容 |
|---|---|
| 削除 | WordState の μ, σ。uncertain プール。sigmaDecay, sigma0, uncertainThreshold パラメータ |
| 追加（導出関数） | uncertaintyWidth(word, currentTime)。観測回数と経過時間から導出。状態を持たない |
| 追加（スケジューラ） | due 判定時の h_effective サンプリング（トンプソンサンプリング） |
| 追加（トリアージ） | urgent プールの多次元優先度スコア |
| 追加（UI） | uncertaintyWidth ベースの信頼性表示 |
| 追加（モード） | 信頼性ベースの確認モード（uncertaintyWidth 降順デッキ） |
| 新パラメータ | uncertaintyBase (0.5), uncertaintyFloor (0.05), staleGrowth, トリアージ重み |

実装順序の提案: (1) 削除とサンプリングを同時に入れて位相同期の解消をシミュレーションで検証 → (2) トリアージ → (3) UI 表示と確認モード。

## 8. 検証計画

### シナリオ E: 位相同期の分散検証

```
条件: 朝集中型学習者（5セッション/朝・6分間隔）× 30日
比較: サンプリングなし（現行） vs h_effective サンプリングあり
測定:
  - due クラスタの大きさ（同一セッションで due になる語数の分布）
  - 「復習なし」発生回数
  - 定着語数・avgH への副作用
期待: クラスタが小さく分散し、復習なしが減少。定着効率は維持
```

### シナリオ F: トリアージ検証

```
条件: 学習中盤の状態（mastered 多数）から3週間離脱 → 復帰
比較: p 昇順のみ vs 多次元トリアージ
測定: mastered 語が復帰後何セッション以内に出題されるか
期待: 投資済み記憶の救済が早まる
```

### 留意点

仮想学習者の「真の忘却」がシステムの推定 h と同一モデルである限り、サンプリングの真価（推定誤りの発見と訂正）は測定できない。シナリオ E/F では、仮想学習者の真の h にシステムの推定とのずれ（例: ±30% の個体差）を導入すること。

## 9. 結語

ベイジアン × SRS の価値は「不確実性を行動に使う」ことにある。それは uncertain プールという箱ではなく、スケジューリングのゆらぎ・トリアージの次元・学習者への正直さとして実装されるべきだった。

σ という状態変数の削除は、ベイジアン構想の放棄ではない。むしろ偽の実装を取り除き、本来の構想を「状態を持たず、導出し、サンプリングする」という軽量な形で回復する第一歩である。

不確実なものはゆらぎ、確認されたものは安定する——この単純な原理が、位相同期を散らし、トリアージに深みを与え、学習者に正直な情報を返す。運まかせは、正しく設計されたとき、最も誠実なスケジューラになる。
