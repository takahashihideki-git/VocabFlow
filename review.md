  総評

  全体としてよく設計されています。半減期モデルと貪欲プール割当はシンプルで一貫しており、death spiral
  防止（hMin）、skipped の逃げ切り防止、Intro→Recognition の gap
  保証など、過去のバグ修正が丁寧に積み上がっています。その上で、改善余地を重要度順に挙げます。

  1. h の更新が実経過時間を無視している（最重要・アルゴリズム本体）

  srs-engine.js:84-89 の h = h × alpha × cardWeight は、前回復習からどれだけ時間が経っていたかを一切考慮しません。p=0.99
  （数分前に見たばかり）での正解も、p=0.5（忘れかけ）での正解も同じ ×2.0 です。これにより:

  - 新語の Intro→Recognition ペアは同一セッション内（数分差）で h が 1.0 → 1.6日 に成長。長期記憶の証拠ゼロでの増加です
  - リトライ正解（2026-04-21 設計変更で processResponse 経由になった）は数分以内に ×0.3 → ×2.0
  が走り、「間違えた直後の短期記憶での正解」が h
  を部分回復させます。これは設計意図どおりですが、回復量が時間情報と無関係です

  FSRS や Duolingo の HLR など現代の SRS は、想起の難しさ（1−p）に応じて報酬をスケールさせます。簡単な改善案: alpha_eff
  = 1 + (alpha − 1) × min(1, deltaT / h) のように deltaT/h でゲインを減衰させるか、(1−p)
  を係数に掛ける。これで「短間隔の正解は h をほぼ伸ばさない」「忘却の縁での正解は大きく伸ばす」が自然に表現されます。sim
  で検証してから導入できる構成（core 共有）なのはこのプロジェクトの強みです。

  2. near_miss / phonetic の扱いが sim と app で乖離している

  2026-04-16 に app は「near_miss = 不正解扱い」へ変更しましたが、これは UI 層のワークアラウンド（スナップショット復元 +
  _markReady('wrong')）で実装されており、エンジン本体 srs-engine.js:13 は今も isCorrect = result !== 'wrong'、つまり
  near_miss を正解（×2.0×0.9=1.8）として処理します。そして sim/sim-runner.js:32-35 は virtual-learner が生成する
  near_miss/phonetic をそのままエンジンに渡しています。

  つまり シミュレータは現在のアプリ挙動をモデルしていません。dictation ステージで sim は楽観的（h が伸びる）、app
  は悲観的（×0.3 で降格）。CLAUDE.md のシミュレーション実績表（Day 189 で
  1000語定着）は実アプリより楽観側にずれている可能性があります。修正案: 「near_miss
  は不正解」というポリシーをエンジン（processResponse）に移し、sim と app が同じ経路を通るようにする。UI
  層のスナップショット復元ロジックも簡素化できます。

  3. mastered の二重定義が残っている（opportunity 事件と同根)

  2026-05-28 の「Wave 2 クリア再発火」は stage と h の二重評価ズレが原因でしたが、同じ構造が models.js:90-92
  に残っています。LearnerState.masteredCount は h >= 14 で数える一方、Wave クリア判定や Word Wave の金色ドットは stage
  === 'mastered' で判定します。降格して stage=dictation だが h=24 の語（まさに opportunity
  の状態）は、ヘッダ統計では「定着」に数えられるのに Wave クリアはブロックするという不整合です。masteredCount を stage
  === 'mastered' に統一することを推奨します（全Waveクリア予測 _computeTide 周りの数値も一貫します）。

  4. Wave 卒業判定に2つの穴がある

  wave-manager.js:95-102 の _isGraduated（90% が h≥8 で activeWaves から除外）:

  - 未導入語の孤児化: 供給ベース解放では、ある wave に最大4語の new 語が残ったまま次 wave が解放されます。その状態で
  wave が卒業条件（90% h≥8、残り new が10%以下なら成立可能）を満たすと activeWaves
  から外れ、getNewWordsFromActiveWaves（wave-manager.js:107-114）は卒業 wave
  を見ないため、その語は永久に導入されず、全Wave
  クリアも永久に達成不能になります。長期離脱からの復帰で復習が飽和し続けるケースで現実に起こり得ます
  - excluded 語が分母に残る: 未学習のまま除外された語は h=0 で分母に入るため、10語以上除外された wave
  は永久に卒業できません(実害は activeWaves が肥大するだけですが)

  修正案: 分母から excluded を外し、非除外の new 語が残っている間は卒業させない(または卒業 wave の残存 new
  語を供給対象に含め続ける)。

  5. ベイズ層（μ/σ）が形骸化している

  - _bayesianUpdate（srs-engine.js:117-130）の μ は正解・不正解どちらでも log(h)
  を代入するだけで、事後分布の更新ではありません。μ はどこからも読まれていない死にフィールドです
  - σ は正解でも不正解でも単調減少します。不正解は「h の推定が外れた」証拠なので、本来 σ は増えるべき場面です
  - 実効性の面でも、σ0=1.0・閾値1.5・成長0.01/日では、uncertain プール入りに「50日以上未復習なのに p がまだ
  0.5〜0.85」という稀な条件が必要で、uncertain プールはほぼデッドコードです。さらに
  - excluded 語が分母に残る: 未学習のまま除外された語は h=0 で分母に入るため、10語以上除外された wave
  は永久に卒業できません(実害は activeWaves が肥大するだけですが)

  修正案: 分母から excluded を外し、非除外の new 語が残っている間は卒業させない(または卒業 wave の残存 new
  語を供給対象に含め続ける)。

  5. ベイズ層（μ/σ）が形骸化している

  - _bayesianUpdate（srs-engine.js:117-130）の μ は正解・不正解どちらでも log(h)
  を代入するだけで、事後分布の更新ではありません。μ はどこからも読まれていない死にフィールドです
  - σ は正解でも不正解でも単調減少します。不正解は「h の推定が外れた」証拠なので、本来 σ は増えるべき場面です
  - 実効性の面でも、σ0=1.0・閾値1.5・成長0.01/日では、uncertain プール入りに「50日以上未復習なのに p がまだ
  0.5〜0.85」という稀な条件が必要で、uncertain プールはほぼデッドコードです。さらに
  _pickSorted（feed-generator.js:319-320）は選別時に使った currentSigma ではなく生の sigma でソートする不整合もあります

  推奨: 思い切って μ/σ/uncertain 機構を削除して簡素化するか、活かすなら「予想外の不正解で σ
  増加」を入れて閾値を現実的にする。中途半端に残っているのが一番メンテコストが高い状態です。

  - spellingFlag は write-only（srs-engine.js:51 でセットされるが、出題・表示のどこからも読まれない）。スペル怪しい語に
  dictation を優先出題する等の活用か、削除を
  - _isPhoneticMatch（srs-engine.js:192-213）: フォールバック条件「編集距離≤2 かつ
  長さ≥4」が広すぎて、大半の2文字タイポが phonetic 判定になります。また input.replace(/ie/, 'ei') は最初の1箇所しか置換

  - Handwrite の常時 perfect: OCR モックが常に perfect を返すため、×2.0×1.6=3.2 倍という最大の h

  - peakH: wave 解放が供給ベースになった現在、core では未使用（sim/scenarios 用に残置と認識)。コメント
