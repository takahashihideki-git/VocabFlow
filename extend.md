予告すべきは「意味のあるセッションが組める時刻」です。due に入る語の時刻を早い順にリストアップして、sessionSize
の半数（10枚）以上が due/urgent になる時刻を表示する。

全学習済み語の bufferedNextReview を昇順ソート
→ 10番目の語の bufferedNextReview が「次のセッション」の予告時刻

4月7日
そうですね。due/urgent + new で半数以上になる時刻です。

new が5語あるなら、due が5語揃う時刻で合計10枚。new が0なら due が10語揃う時刻。

newCount = アクティブウェーブ内の未学習語数（上限 maxNewPerSession）
needed = sessionSize / 2 - newCount
→ needed 番目の語の bufferedNextReview が予告時刻
new が多ければ早く、new が枯渇していれば due 待ちで遅くなる。自然ですね。

# 3
 これはようするに同じ半減スケジュールの単語が重なりすぎるという問題ですよね。

19:06
まさに位相同期の再発ですね。シミュレーターで発見したのと同じ構造の問題が、実ユーザーの行動パターンで起きている。

初日にまとめて導入した語が全部 h = 1.0 から始まるので、全語の due タイミングが揃ってしまう。全員が同時に due になり、同時に復習され、同時に h が伸び、また同時に due になる。このクラスタが解消されない。

分散消化だと導入時刻がずれるので、自然に位相がばらける。だからうまくいく。

解決策は、同時に導入された語の h の初期値に意図的なばらつきを持たせることかもしれません。全語一律 h0 = 1.0 ではなく、h0 を 0.8〜1.2 の範囲でランダムに振る。すると due タイミングが数時間ずつずれて、クラスタが自然に分散する。

実際詰め込みすぎ、は記憶維持能力にプレッシャーをかけるわけだから。

19:07
その通りです。実際の人間なら、1日に150語詰め込んだら1語あたりの記憶強度は当然落ちる。でもアルゴリズムは全語一律 h0 = 1.0 で扱っている。

現実には、1セッション目に出会った語と10セッション目に出会った語では、後者の方が記憶に残りにくい。脳が疲れているし、前の語との干渉もある。

つまり h0 を固定値ではなく、その日の累積新語数に応じて減衰させるのが現実に近い。

h0_adjusted = h0 × decay^(todayNewCount)
今日の1語目は h0 = 1.0、10語目は h0 = 0.8、30語目は h0 = 0.5、のように。たくさん詰め込むほど1語あたりの初期記憶強度が下がり、due が早く来る。結果として復習が増え、新語の投入ペースに自然にブレーキがかかる。

これならmax_active_waves のような人為的な上限ではなく、認知負荷の現実をモデルに組み込む形でペースが制御される。学習者へのリスペクトを保ったまま。

 これは大事な洞察ですね。

19:08
ありがとうございます。これを仕様に入れるなら、SRSウェーブ方式の設計原則に新しい柱が加わります。

「認知負荷の制御はパラメータの上限値ではなく、モデルの中に組み込む」

max_active_waves は人為的な天井でした。h0 の減衰は認知の現実をモデル化している。同じ「ペース制御」でも、設計思想が根本的に違います。

パラメータとしては：

h0DecayBase = 0.98  // 1語追加ごとの減衰率
h0Min = 0.3         // h0 の下限（これ以下にはしない）
今日の新語数が0語なら h0 = 1.0、10語なら h0 = 0.82、30語なら h0 = 0.55、50語なら h0 = 0.36。

副次効果として位相同期も自然に解消されます。同じ日に導入された語でも h0 が微妙に異なるので、due タイミングがばらける。
